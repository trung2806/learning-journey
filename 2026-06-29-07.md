## Thay đổi kỹ thuật

### `_DAEMON_VERSION` constant

```python
# ==============================================================================
# CONFIGURATION & METADATA CONSTANTS (SINGLE SOURCE OF TRUTH)
# ==============================================================================
_DAEMON_VERSION = "Day 47"

# ... (sau đó trong main_loop)
logging.info(f"🚀 Khởi động SR Linux Monitor Fleet ({_DAEMON_VERSION})...")
```

**Tại sao string literal hardcoded gây regression liên tục?**

String `"Day 41"` nằm sâu trong thân hàm `main_loop`, cách ~200 dòng từ đầu file. Khi user copy code để modify cho ngày mới, flow thường là: đọc đầu file → thêm feature → xem lại import → không scroll đủ sâu để thấy startup log. Kết quả: bị drop 5 lần liên tiếp (D43→D47).

Vấn đề không phải "nhớ hơn" mà là **locality** — thông tin cần update (số ngày) nằm quá xa điểm tự nhiên người đọc update (đầu file, nơi constants và metadata tập trung).

`_DAEMON_VERSION` di chuyển thông tin này về đầu file nơi constants sống, group với `POLL_TIMEOUT_SECONDS`, `_INFRA_ERROR_STATUSES`, `DEFAULT_THRESHOLDS`. Lần sau khi tạo ngày mới, người chỉ cần update 1 dòng ở đầu file.

**Pattern tổng quát: Single Source of Truth cho metadata**

Bất cứ khi nào một giá trị được dùng ở nhiều nơi hoặc có nguy cơ drift, extract ra constant có tên rõ ràng. Không phải vì DRY (chỉ dùng 1 chỗ) mà vì **discoverability** — người maintain biết chính xác nơi cần update.

Tương tự trong production code:
- `VERSION = "1.2.3"` trong `__init__.py` thay vì hardcode trong setup.py + pyproject.toml + changelog
- `BASE_URL = "https://api.example.com"` thay vì lặp trong mỗi endpoint function
- `TIMEOUT_SECONDS = 30` thay vì magic number rải rác

**Tại sao prefix `_`?**

Convention Python: `_name` = module-internal, không export ra ngoài khi `from module import *`. `_DAEMON_VERSION` là metadata nội tại của daemon, không cần expose cho test hoặc caller bên ngoài. Khác với `DEFAULT_THRESHOLDS` (không có `_`) vì `test_pipeline.py` import nó trực tiếp.

## Không thêm test

`_DAEMON_VERSION` chỉ ảnh hưởng log string — không có behavior change có thể assert. Test `test_sigusr1_does_not_stop_daemon` và các test integration đã cover startup sequence gián tiếp. Test riêng cho log content sẽ là over-testing (assert implementation detail, không phải behavior).

## Ruling

- **Single Source of Truth cho metadata**: giá trị cần update theo thời gian (version, URL, timeout) sống ở một chỗ, tên rõ ràng, gần đầu file nơi constants tập trung. Không duplicate trong string literal.
- **Locality over memory**: giải pháp tốt không yêu cầu người maintain nhớ nhiều hơn. Đặt thông tin gần điểm update tự nhiên của nó.
- **Prefix `_` cho module-internal constant**: phân biệt API public (`DEFAULT_THRESHOLDS`) vs implementation detail (`_DAEMON_VERSION`, `_INFRA_ERROR_STATUSES`). Không phải bắt buộc nhưng là tín hiệu rõ ràng về intent.

## Glossary mới

- **Single Source of Truth (SSOT)**: nguyên tắc thiết kế — mỗi piece of knowledge chỉ có một representation duy nhất authoritative trong hệ thống. Thay đổi ở một nơi, không có nơi nào khác cần sync thủ công.
- **Magic number / magic string**: giá trị hardcoded trong code không có tên rõ ràng, không giải thích tại sao có giá trị đó. `"Day 41"` là magic string. `_DAEMON_VERSION = "Day 47"` là named constant.
- **Locality of information**: thông tin nên nằm gần nơi nó được dùng hoặc nơi người maintain tự nhiên tìm kiếm nó. Violate locality → thông tin bị bỏ sót khi maintain.
