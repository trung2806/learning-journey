## Quyết định mở session: connection lifecycle = open-once + inject conn

Chốt **open-once + inject conn**: `build_row` sở hữu vòng đời SSH qua context manager (`with ConnectHandler(...) as conn:`), mỗi `get_*(conn)` nhận conn đã mở. Lý do: triệt tiêu rủi ro vận hành AAA lockout / session limit trên gear thật, giữ vững Engine/IO split, và cô lập + kiểm thử tường minh các kịch bản partial-failure ngay tại `build_row`.

Hai điểm tinh chỉnh nhận ra khi khung lại fork:

1. Cái mình từng liệt kê là *cost* của open-once ("build_row phải quản lý vòng đời conn") thật ra là *design win*, không phải giá phải trả. Inject conn làm `get_version` mock được tầm thường (truyền mock conn, stub `send_command`), đúng pattern Day 8/12. Per-call (mở `ConnectHandler` bên trong get_version) nhét resource acquisition vào trong hàm, khó mock và khó test nhánh fail. Open-once đồng nhất với Engine/IO split chứ không nghịch nó.
2. Per-call có một operational failure mode không liệt kê lúc đầu: N handshake SSH/AAA cho N cột. Nhiều NOS có session limit, handshake lặp chậm và có thể trip rate-limit hoặc AAA lockout. "N lần SSH" là rủi ro vận hành, không chỉ kém elegant.

Per-call *có* một lợi thế thật là fault isolation (session get_version chết không kéo get_cpu chết theo). Nhưng đó chính là cái buộc xử lý partial-failure ở build_row, và partial-failure lại tách đúng hai trục mình build từ Day 15: transport failure là một trục, parse failure là trục khác.

## Kiến trúc 2 trục lỗi: Transport (hạ tầng) vs Parse (dữ liệu)

`build_row` là orchestration layer, nơi DUY NHẤT quản lý sinh tử của connection, và phân định rạch ròi:

- **Trục Transport, tầng khởi tạo** (handshake / AAA): `NetmikoTimeoutException`, `NetmikoAuthenticationException` bắt tại `with`, dán nhãn `status = "Unreachable"`, `[INIT TRANSPORT FAIL]`.
- **Trục Transport, mid-call** (đứt cáp / lệnh treo lúc đang chạy): `OSError`, `NetmikoTimeoutException`, `ReadTimeout` bắt ở inner clause, dán nhãn `status = "Disconnected"`, `version = "Conn Drop"`, `[MID-CALL DROP]`.
- **Trục Parse** (thiết bị phản hồi nhưng dữ liệu lỗi): `VersionParseError` định tuyến theo `reason`. `empty` là boundary (thiết bị sống, lệnh chạy, không ra chữ) dán `[BOUNDARY]`; `malformed` / `schema` dán `[PARSE ERROR]`.
- **Bug code lạ** (không thuộc hai trục trên): outer `except Exception` dán `status = "Setup Error"`, `[UNEXPECTED SYSTEM ERROR]`, GIỮ NGUYÊN `traceback.format_exc()` để dev vào sửa ăn ngay.

Điểm thiết kế đắt nhận ra: `NetmikoTimeoutException` xuất hiện ở CẢ inner mid-call lẫn outer init clause, disambiguate thuần bằng vị trí trong vòng đời. Cùng một exception type, hai nhãn operator-meaningful tùy nó fire ở đâu. Timeout lúc handshake thành "Unreachable", timeout lúc send_command thành "Conn Drop".

## srl_app.py (bản final, đã verify xanh)

```python
import traceback
from netmiko import ConnectHandler
from netmiko.exceptions import (
    NetmikoTimeoutException,
    NetmikoAuthenticationException,
    ReadTimeout,  # Khóa chặt mảnh ghép thiếu của mid-call timeout
)
from srl_engine import parse_version, VersionParseError


def get_version(ssh_conn) -> str:
    """IO Bound: gửi lệnh và đẩy dữ liệu thô cho Engine xử lý."""
    raw_output = ssh_conn.send_command("show version | as json")
    return parse_version(raw_output)


def build_row(device_config: dict) -> dict:
    """
    Orchestration Layer: quản lý sinh tử của connection.
    Phân tách rạch ròi 2 trục lỗi: Transport (hạ tầng) và Parse (dữ liệu).
    """
    row = {
        "host": device_config.get("host"),
        "version": "N/A",
        "status": "Unknown",
        "error_log": None,
    }

    try:
        # TRỤC 1: TRANSPORT FAILURE TẦNG KHỞI TẠO (handshake / AAA lockout)
        with ConnectHandler(**device_config) as conn:
            row["status"] = "Reachable"

            try:
                row["version"] = get_version(conn)

            # TRỤC 2: PARSE FAILURE (thiết bị phản hồi nhưng dữ liệu lỗi cấu trúc)
            except VersionParseError as pe:
                row["version"] = "Parse Fail"
                if pe.reason == "empty":
                    row["error_log"] = "[BOUNDARY] Lệnh trả về rỗng. Thiết bị sống nhưng CLI lỗi hoặc treo."
                else:
                    row["error_log"] = f"[PARSE ERROR] Reason: {pe.reason} | Chi tiết: {pe}"

            # TRỤC 1 TIẾP DIỄN: TRANSPORT FAILURE MID-CALL (đứt cáp HOẶC lệnh treo => ReadTimeout)
            except (OSError, NetmikoTimeoutException, ReadTimeout) as mid_err:
                row["version"] = "Conn Drop"
                row["status"] = "Disconnected"
                row["error_log"] = f"[MID-CALL DROP] Kết nối bị ngắt giữa chừng hoặc lệnh treo: {str(mid_err)}"

    except (NetmikoTimeoutException, NetmikoAuthenticationException) as init_err:
        row["status"] = "Unreachable"
        row["error_log"] = f"[INIT TRANSPORT FAIL] Không thể mở phiên SSH: {str(init_err)}"

    except Exception as general_err:
        # Biên ngoài cùng CHỈ ôm bug code hệ thống, giữ nguyên stack trace
        row["status"] = "Setup Error"
        row["version"] = "N/A"
        row["error_log"] = (
            f"[UNEXPECTED SYSTEM ERROR] Lỗi code hệ thống: {str(general_err)}\n"
            f"Traceback:\n{traceback.format_exc()}"
        )

    return row
```

## Defect bắt trong review #1: catch-all đội lốt

Bản draft đầu có `except (IOError, OSError, Exception) as mid_err:`. Đây là bare `except Exception` đội lốt, vi phạm chính glossary #25 của mình ("catch all che giấu bug"):

- Python 3: `IOError` là alias của `OSError`, nên `(IOError, OSError)` đã redundant (flake8-bugbear B014 bắt được).
- Có `Exception` trong tuple làm cả cụm collapse về `except Exception` (`OSError ⊂ Exception`), hai type cụ thể kia chỉ là trang trí.

Tệ hơn smell: đây là defect conflation Day 14 tái sinh ở tầng orchestrator. Bất kỳ exception non-transport nào (bug trong parse_version, `AttributeError`, typo thành `NameError`) bị dán nhãn `[MID-CALL DROP]`. Operator đi truy network khi thủ phạm là bug code. Trớ trêu là mình đã có home đúng cho bug lạ (bucket outer `[UNEXPECTED SYSTEM ERROR]`), nhưng inner catch-all chặn họng nó trước.

Fix: gọi tên đúng transport exception fire được mid-call, bỏ `Exception`, để mọi thứ ngoài set đó propagate lên outer handler.

## Defect bắt trong review #2: lỗ transport set, ReadTimeout không phải subclass của NetmikoTimeoutException

Fact check trên netmiko 4.7.0 thật:

```
ReadTimeout subclass của NetmikoTimeoutException? False
MRO: ReadTimeout -> ReadException -> NetmikoBaseException -> Exception -> BaseException -> object
```

`send_command` raise `ReadTimeout` khi không thấy CLI prompt trong read_timeout, và đây là manifestation phổ biến NHẤT của "thiết bị reachable nhưng lệnh treo". Với clause cũ `(OSError, NetmikoTimeoutException)`, một mid-call `ReadTimeout` lọt qua inner clause, rơi xuống outer `except Exception`, bị dán "Setup Error". Đó là transport failure mặc áo code-bug, conflation chạy ngược chiều defect #1. Fix: thêm `ReadTimeout` vào inner clause (hoặc `ReadException` nếu muốn ôm cả read-side family).

## Verify RED-on-defect (2 pin được chứng minh không vacuous)

Không chỉ chạy xanh, mà revert defect rồi xem pin có đỏ đúng chỗ:

- Negative pin revert về `(IOError, OSError, Exception)`: đỏ với `assert 'Conn Drop' != 'Conn Drop'`. Tức bug code bị catch-all nuốt và dán "Conn Drop" đúng như cảnh báo.
- ReadTimeout pin revert về `(OSError, NetmikoTimeoutException)`: đỏ với `assert 'Setup Error' == 'Disconnected'`. Tức không có ReadTimeout thì mid-call hang bị đá lên Setup Error.

Cả hai xanh trên bản fix, đỏ trên bản defect. Pin bite thật.

## Bài học hôm nay

1. **Dispatch-contract principle nổi lại ở tầng orchestrator, hai lần, ngược chiều nhau.** Catch-all đẩy code-bug thành "Conn Drop"; thiếu ReadTimeout đẩy transport hang thành "Setup Error". Cùng một conflation. Fix cả hai đều là: gọi tên chính xác exception thuộc từng trục, để phần còn lại propagate về đúng bucket. Except-ladder của `build_row` CHÍNH LÀ kỷ luật `reason` của engine, nâng lên một tầng. Engine định tuyến trục dữ liệu bằng `reason`; orchestrator định tuyến trục transport bằng exception type. Routing bằng danh tính chính xác, hai tầng.
2. **Một routing pin chỉ đáng giá bằng cái nó CẤM.** Chứng minh RED-on-defect trước khi tin nó xanh. Đây là kỷ luật false-green nâng thành thao tác: pin nào canh invariant routing/conflation thì xứng đáng một lần revert-and-watch-RED.
3. **Cùng một exception, hai nhãn operator-meaningful theo vị trí vòng đời.** NetmikoTimeoutException ở init clause vs mid-call clause.
4. **Đóng tại ranh giới khác defer.** Backlog việc-đã-xong = 0 lúc đóng (defect #1 discharged + pin, transport set complete + pin). Cái mang sang là việc-chưa-bắt-đầu.

## Glossary thêm

- **Exception-type-as-transport-dispatch** (đối xứng với reason-as-data-dispatch): routing hai tầng, engine dùng `reason` cho trục dữ liệu, orchestrator dùng exception type cho trục transport.
- **Disguised catch-all**: `(IOError, OSError, Exception)` collapse về bare `Exception` (IOError là alias OSError trong Py3; OSError ⊂ Exception). flake8-bugbear B014 bắt redundancy.
- **ReadTimeout vs NetmikoTimeoutException**: KHÔNG phải subclass. MRO đi qua `ReadException -> NetmikoBaseException`. `send_command` raise `ReadTimeout` khi không thấy prompt. Muốn ôm trục transport mid-call đầy đủ phải kể tên `ReadTimeout` riêng.
- **RED-on-defect verification**: pin canh invariant routing/conflation phải qua một lần revert-defect-xem-đỏ trước khi tin.
- **Inject-conn (dependency injection) cho IO shell**: resource do orchestrator sở hữu qua context manager, worker chỉ mượn, nhờ vậy mock tầm thường.
