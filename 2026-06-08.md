### Pin biên — monkeypatch-time deterministic (rep test mới)
Bẫy đã tránh: nếu set mtime "đúng 30 ngày" rồi assert bị loại theo kiểu naive, test sẽ FAIL — vì `now = time.time()` gọi BÊN TRONG hàm sau khi set vài ms, làm `age_seconds` hơi lớn hơn ngưỡng → file bị include, nhìn như bug nhưng chỉ là timing gap. Thuốc giải: đóng băng clock.
```python
import os
import time
from pathlib import Path

import find_stale


def test_find_old_files_knife_edge(monkeypatch, tmp_path):
    """Knife-edge, deterministic bang cach dong bang time.time().

    - File dung khit nguong (age == n_seconds): bi LOAI (strict `>`).
    - File gia hon nguong 1 giay: duoc GIU.
    """
    FIXED_NOW = 1770000000.0
    monkeypatch.setattr(find_stale.time, "time", lambda: FIXED_NOW)
    days_threshold = 30
    n_seconds = days_threshold * 86400

    edge_file = tmp_path / "edge_exactly_30d.log"
    edge_file.write_text("File vua tron 30 ngay.")
    mtime_edge = FIXED_NOW - n_seconds            # age == n_seconds
    os.utime(edge_file, (mtime_edge, mtime_edge))

    stale_file = tmp_path / "stale_30d_plus_1s.log"
    stale_file.write_text("File gia hon nguong dung 1 giay.")
    mtime_stale = FIXED_NOW - n_seconds - 1        # age == n_seconds + 1
    os.utime(stale_file, (mtime_stale, mtime_stale))

    result = find_stale.find_old_files(tmp_path, days_threshold)
    result_paths = [path for path, age in result]

    assert stale_file in result_paths, "File gia hon nguong 1s phai duoc GIU."
    assert edge_file not in result_paths, "File khit nguong (chua vuot) phai bi LOAI."
    assert len(result) == 1
    assert result[0][1] == 30   # pin luon logic lam tron cot tuoi
```
- Vì sao deterministic được: `FIXED_NOW` nguyên giây nên `os.utime` round-trip không mất precision, `st_mtime` trả đúng số; knife-edge `>` pin được vì `now` đứng yên.
- Test cũ (Day 12) chạy lại → **xanh giả**: nó viết theo contract đã drift, assert quá lỏng không bắt được đổi shape. Đã **xoá hẳn**, không để cạnh test đúng. Một test xanh-giả tệ hơn không có test: nó cho confidence mà pin con số 0. Một file, một invariant.

## Version qua `| as json` — structured tại nguồn (chốt parse strategy)
Hai lựa chọn nằm hai phía một trục: TextFSM/ntc-templates là parse-after-the-fact (họ hàng regex, fragile, template theo từng lệnh/vendor) vs `| as json` là structured-tại-nguồn. Chọn `| as json`.

Output thật bắt được từ srl1 (v26.3.2), dùng làm fixture chuẩn:
```json

    "Hostname": "srl1",
    "Chassis Type": "7220 IXR-D2L",

Cùng một trục — để parser thật ăn structured data tại nguồn, đừng bolt string-bashing lên sau. Lesson regex-over-match áp được sang một domain mới (NOS output thay vì config file), tự phát. Đây là consolidation, đáng giá hơn việc chạy trước lịch.

## Engine/IO split — srl_driver.py
`get_version_from_device` nhận `ssh_conn` như tham số (dependency injection), KHÔNG tạo connection bên trong. Đó là lý do mock chạy gọn mà không phải patch `ConnectHandler`. (Day 14 standalone proof tạo connection trong hàm; bản này đẩy nó ra ngoài.)
```python
import json
# ==========================================
# ENGINE: Pure Function (Không side-effect, dễ test)
# ==========================================
def parse_version(raw_json: str) -> str:
    """
    Nhận chuỗi JSON thô từ thiết bị và trích xuất version.
    Không quan tâm dữ liệu từ đâu ra (SSH, File, API...).
    """
    if not raw_json or not raw_json.strip():
        raise ValueError("Thiết bị trả về chuỗi rỗng hoặc không hợp lệ.")

    data = json.loads(raw_json)

    try:
        return data["basic system info"]["Software Version"]
    except KeyError as e:
        raise KeyError(f"Cấu trúc JSON thay đổi, thiếu key: {e}")
# ==========================================
# IO SHELL: Đảm nhận kết nối vật lý
# ==========================================
def get_version_from_device(ssh_conn) -> str:
    """
    IO Bound: Chỉ chịu trách nhiệm bắn lệnh qua SSH và đẩy kết quả cho Engine parse.
    """
    raw_output = ssh_conn.send_command("show version | as json")
    return parse_version(raw_output)
```

## Test — test_srl_driver.py (5 test, đã verify xanh)
Hai fix so bản đầu: (1) `test_parse_version_invalid_json` đặt tên SAI — `'{"wrong_key": {}}'` là JSON hợp lệ sai cấu trúc, nó test nhánh missing-key chứ không phải invalid JSON; đổi tên thành `test_parse_version_missing_key`. (2) Nhánh invalid JSON THẬT (chuỗi không-rỗng-không-phải-JSON, failure khả dĩ nhất ngoài đời) chưa có test → thêm `test_parse_version_not_json`.

Một test đặt tên sai cùng lớp lỗi với journal note: nó khẳng định một coverage nó không có, nói dối người đọc sau (có thể là chính mình ba tuần nữa) rằng "cái này được đảm bảo".
```python
import json

import pytest
from unittest.mock import MagicMock

from srl_driver import parse_version, get_version_from_device

# Cục JSON thật bốc được từ srl1 (v26.3.2) làm mẫu chuẩn
RAW_JSON_SAMPLE = """{
  "basic system info": {
    "Hostname": "srl1",
    "Chassis Type": "7220 IXR-D2L",
    "Part Number": "Sim Part No.",
    "Serial Number": "Sim Serial No.",
    "System HW MAC Address": "1A:02:00:FF:00:00",
    "OS": "SR Linux",
    "Software Version": "v26.3.2",
    "Build Number": "426-g2b38957bbca",
    "Architecture": "aarch64",
    "Last Booted": "2026-06-06T04:19:50.152Z",
    "Total Memory": "7915708 kB",
    "Free Memory": "4329488 kB"
  }
}"""


# --- ENGINE: parse_version thuần ---

def test_parse_version_success():
    assert parse_version(RAW_JSON_SAMPLE) == "v26.3.2"


def test_parse_version_missing_key():
    # JSON HỢP LỆ nhưng sai cấu trúc (thiếu key) -> KeyError
    valid_json_wrong_shape = '{"wrong_key": {}}'
    with pytest.raises(KeyError):
        parse_version(valid_json_wrong_shape)


def test_parse_version_empty_input():
    # Input rỗng/whitespace -> ValueError (guard tường minh của engine)
    with pytest.raises(ValueError):
        parse_version("")


def test_parse_version_not_json():
    # Chuỗi không-rỗng-không-phải-JSON (vd dòng lỗi CLI) -> JSONDecodeError.
    # Lưu ý JSONDecodeError là subclass của ValueError, nên hiện tại empty và
    # garbage CHƯA phân biệt được ở tầng ValueError -> quyết định tách 3 failure
    # mode hay không để Day 15.
    with pytest.raises(json.JSONDecodeError):
        parse_version("Error: unknown command")


# --- IO INTEGRATION: mock send_command ---

def test_get_version_from_device_flow():
    mock_ssh_conn = MagicMock()
    mock_ssh_conn.send_command.return_value = RAW_JSON_SAMPLE

    version = get_version_from_device(mock_ssh_conn)

    assert version == "v26.3.2"
    # pin luôn câu lệnh gửi đi = pin contract IO, không chỉ giá trị trả về
    mock_ssh_conn.send_command.assert_called_once_with("show version | as json")
```
Verify: `pytest test_srl_driver.py -v` → 5 passed.

## Glossary thêm
- **Dependency Injection (DI)**: truyền phụ thuộc (vd `ssh_conn`) vào hàm/đối tượng thay vì tạo bên trong. Cho phép test mock phụ thuộc mà không patch điểm tạo. Là lý do `get_version_from_device(ssh_conn)` test được bằng `MagicMock`.
- **`json.loads` như structured-tại-nguồn**: thiết bị emit JSON (`| as json`) → để `json.loads` ăn thẳng, không TextFSM/regex parse string trần. Cùng trục `yaml.safe_load` vs regex (port_check).
- **monkeypatch `time.time` (frozen clock)**: `monkeypatch.setattr(mod.time, "time", lambda: FIXED)` đóng băng đồng hồ → pin được biên thời gian knife-edge deterministic, tự revert sau test.
- **`MagicMock(...).return_value` + `assert_called_once_with`**: dựng object giả có hành vi (`send_command.return_value = ...`) và assert nó được gọi đúng một lần với đúng argument → pin cả contract IO.
- **`json.JSONDecodeError` là subclass của `ValueError`**: chuỗi non-JSON raise nó, và nó lọt vào `pytest.raises(ValueError)`. Hệ quả: empty và garbage hiện gộp một ở tầng `ValueError`.
