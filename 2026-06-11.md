## Quyết định 1: tách config = hai dict riêng từ inventory

Chốt phương án (a): `{"connection": {...}, "meta": {"platform": ..., "site": ...}}`. Loại (b) — dict phẳng rồi `build_row` tự lọc key — vì nó buộc `build_row` sở hữu blacklist/whitelist tham số Netmiko: Netmiko thêm param mới hoặc report thêm tag mới là `build_row` gãy. Đó là leaky abstraction. Tách `connection` cô lập đúng phần thuộc hạ tầng/giao thức, `meta` thuần cho nghiệp vụ/hiển thị/dispatch. Map khớp với open-once + inject conn của Day 16.

Forward risk Day 16 đóng luôn theo: `ConnectHandler(**connection_kwargs)` giờ chỉ ăn kwargs sạch, tag `platform` sống bên `meta`, không bao giờ rò vào constructor.

## Quyết định 2: registry ở module dispatch riêng

Chốt: `dispatch.py` tách khỏi cả `srl_engine` lẫn `srl_app`. Lý do: `srl_engine` là functional core sạch chuyên JSON Nokia, nhét schema text Cisco vào là làm bẩn cohesion; `srl_app` là orchestrator quản vòng đời conn, không gánh mapping lệnh. Registry là Open-Closed: thêm platform mới = viết hàm parse thuần ở module mới + khai báo vào registry, không chạm luồng chính.

```python
REGISTRY = {
    "nokia_srl": {
        "version": {"cmd": "show version | as json", "parser": parse_srl_version},
        "cpu": {"cmd": "show system resource cpu", "parser": lambda raw: "12%"},  # Stub tường minh
    },
    "cisco_ios": {
        "version": {"cmd": "show version", "parser": parse_ios_version},
        "cpu": {"cmd": "show processes cpu sorted | exclude 0.00%", "parser": lambda raw: "5%"},  # Stub tường minh
    },
}
```

`get_dispatcher(platform, metric)` trả `(cmd, parser)`, raise `ValueError("[CONFIG BUG] ...")` cho platform/metric chưa đăng ký — propagate lên outer handler thành Setup Error kèm traceback, đúng bucket (lỗi cấu hình/code, không phải lỗi thiết bị).

`execute_metric(ssh_conn, platform, metric)` là IO shell duy nhất: lấy cặp từ registry, bắn lệnh, đẩy raw cho parser. `get_version` hardcode lệnh đã chết — dispatch off tag mà không biết platform nào tồn tại.

## parse_ios_version — parser thật, fail-closed

Draft đầu là **parser giả**: hardcode `return "15.2(4)M3"` khi thấy "Cisco IOS Software", và nhánh không khớp trả `"Unknown IOS Version"` như giá trị thành công (fail-open: row hiện Reachable, error_log None, version là chuỗi rác). Pin 2 cũ xanh chỉ vì assert khớp hằng số hardcode — test xanh-giả đúng định nghĩa Day 11, pin con số 0.

Bản fix:

```python
def parse_ios_version(raw_text: str) -> str:
    """Parser text thuần bằng Regex cho Cisco IOS - Không Fail-Open."""
    if not raw_text or not raw_text.strip():
        raise VersionParseError("empty", "Cisco IOS phản hồi chuỗi trống từ thiết bị.")
    match = re.search(r"Version ([^,\s]+)", raw_text)
    if not match:
        raise VersionParseError("schema", "Dữ liệu thô không khớp định dạng chuẩn của Cisco (Thiếu token 'Version <xyz>').")
    return match.group(1)
```

Note nguyên tắc: text parse ở đây KHÔNG vi phạm structured-tại-nguồn — IOS classic không có JSON path, nguồn không emit structured thì parse-after-the-fact là hợp lệ, miễn fail-closed. Trục structured-tại-nguồn áp khi nguồn CÓ structured option (như `| as json` của SRL).

## build_row — per-cell ladder + errors_collected

Conflation tái sinh **lần 3**: draft cell 2 (CPU) dùng `except Exception as cpu_err` — sau Day 14 (engine, `except ValueError` gộp empty/garbage) và Day 16 (orchestrator, catch-all đội lốt). Hậu quả kép: bug code trong cpu parser bị dán `[CPU_PARSE_ERR]` thay vì văng lên Setup Error; transport drop giữa lệnh cpu bị dán Parse Fail thay vì Disconnected. Pin 7 cũ không bắt được vì side_effect nổ ngay call đầu, không bao giờ chạm cell 2.

Fix: cell 2 mang đúng ladder của cell 1 — `VersionParseError` theo reason, transport tuple `(OSError, NetmikoTimeoutException, ReadTimeout)` với early return (đứt cáp thì không cào metric tiếp), mọi thứ khác propagate lên outer.

`errors_collected: list` thay vì ghi đè `error_log` đơn: version fail + cpu fail cùng phiên thì cả hai message sống, join lúc đóng; ca version-parse-fail-rồi-cpu-transport-drop giữ luôn log trước đó. Giết note overwrite của review.

Config guard đầu hàm: thiếu `connection` hoặc `platform` → return sớm `Setup Error` + `[CONFIG ERROR]`, không chạm ConnectHandler.

**Quyết định có chủ đích — hợp nhất nhãn:** nhãn `[BOUNDARY]` riêng cho reason `empty` của Day 16 bỏ, empty giờ đi chung `[VERSION_PARSE_ERR]: [EMPTY] ...`. Thông tin không mất (reason tự mang nhãn trong message qua `reason.upper()`), operator vẫn route được. Ghi rõ ở đây để sau này đọc lại không tưởng là drift.

## Glossary thêm

- **Fail-open vs fail-closed (parser)**: fail-open trả sentinel/giá trị mặc định khi không parse được → lỗi tàng hình thành công; fail-closed raise exception đúng reason. Tool báo cáo/an ninh bắt buộc fail-closed.
- **Leaky abstraction (config filtering)**: tầng orchestrator phải biết danh sách tham số hợp lệ của thư viện bên dưới để lọc dict → abstraction rò. Fix bằng tách contract dữ liệu tại nguồn (inventory), không lọc tại điểm dùng.
- **Open-Closed qua registry**: hành vi mới (platform/metric) thêm bằng đăng ký data, không sửa luồng chạy. `get_dispatcher` là điểm đóng, `REGISTRY` là điểm mở.
- **`side_effect` dạng list (mock tuần tự)**: `mock.side_effect = [val1, Exc()]` — call 1 trả val1, call 2 raise. Cách duy nhất đánh trúng cell sau khi cell trước phải thành công.
- **Exception home theo contract, không theo nơi sinh**: exception dùng chung bởi nhiều module ngang hàng phải sống ở module trung lập, không ở module của instance đầu tiên sinh ra nó.
