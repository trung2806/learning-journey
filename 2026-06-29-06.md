## Thay đổi kỹ thuật

### 1. GUARD CHECKLIST comment

```python
# ==============================================================================
# GUARD CHECKLIST: KHÔNG ĐƯỢC DROP BLOCK NÀY TRONG CÁC LẦN REFACTOR TIẾP THEO
# 1. `fut is not stop_task`: identity check — tránh xử lý nhầm stop_task là network task.
# 2. `tasks.discard(fut)`: O(1) removal từ set — discard (không phải remove) để tránh
#    KeyError nếu invariant asyncio.wait bị vi phạm do bug trong tương lai.
# ==============================================================================
if fut is not stop_task:
    tasks.discard(fut)
```

**Tại sao comment này cần thiết?**

Pattern `fut is not stop_task` + `tasks.discard(fut)` đã bị drop 3 lần liên tiếp (D43 fix → D44 drop → D45 drop → D46 drop lại) khi user copy/paste và modify code. Không phải lỗi logic — là lỗi context switch: khi viết code mới, không nhớ tại sao hai dòng này quan trọng.

Giải pháp đúng không phải "nhớ hơn" mà là **đặt lý do ngay tại điểm rủi ro**. Comment này không giải thích WHAT (code đã tự nói điều đó) mà giải thích WHY — tại sao `discard` chứ không `remove`, tại sao `is not` chứ không `!=`. Đây là loại comment đáng viết theo CLAUDE.md rule.

**Bài học thiết kế:** code tốt không chỉ đúng mà còn **resistance to future regression**. Có hai cách: test coverage (behavior được assert sẽ fail khi drop) và invariant documentation (comment giải thích constraint không hiển nhiên). Cả hai bổ sung cho nhau.

**Tại sao comment gốc user viết SAI?**

User viết:
```
# 1. Kiểm tra fut != stop_task
# 2. tasks.remove(fut) đạt O(1)
```

Comment này document đúng ý định nhưng sai pattern — `!=` thay vì `is not`, `remove` thay vì `discard`. Một comment document sai pattern thì không những không giúp mà còn perpetuate bug cho người đọc. Fix: comment phải document pattern ĐÚNG với lý do đủ để phân biệt.

### 2. `isinstance(raw_data, dict)` defensive guard

```python
# Trước D46:
healthz = raw_data.get("healthz")

# Sau D46:
healthz = raw_data.get("healthz") if isinstance(raw_data, dict) else None
```

`_wrapped_poll` có type annotation `-> Tuple[str, Dict[str, Any]]` nhưng annotation không được enforce ở runtime. Nếu trong tương lai có bug làm `_wrapped_poll` trả về `(host, None)` hoặc `(host, "error_string")`, dòng `raw_data.get("healthz")` sẽ crash với `AttributeError`.

`isinstance(raw_data, dict)` guard bắt trường hợp này tại điểm dùng, không phải tại điểm tạo ra. Pattern này được gọi là **defensive programming at use site** — không tin tưởng type annotation, validate trước khi dùng API của type.

Trade-off: thêm một check không cần thiết trong happy path. Acceptable vì:
- Overhead không đáng kể (isinstance rất nhanh)
- Fail fast với `None` thay vì crash với `AttributeError` ở dòng tiếp theo
- Self-documenting: reader biết `raw_data` có thể không phải dict

### 3. `done, _ = asyncio.wait(...)` cleanup

```python
# Trước: done, pending = ...  (pending không dùng)
# Sau:   done, _ = ...
```

`_` là Python convention cho "giá trị được unpack nhưng không dùng". Không phải micro-optimization — là tín hiệu explicit cho reader: đây là intentional discard, không phải quên. Không dùng mà để tên `pending` gây hiểu nhầm: có phải pending dùng ở dưới không? Reader phải scan thêm để xác nhận.

## Ruling

- **Comment giải thích WHY, không WHERE/WHAT**: "KHÔNG ĐƯỢC DROP BLOCK NÀY" + lý do tại sao từng dòng tồn tại. Code mà không có lý do dễ bị xóa khi refactor. Code có lý do rõ thì người refactor phải explicitly justify việc xóa.
- **Comment sai thì tệ hơn không có comment**: comment document wrong pattern (`remove`, `!=`) dạy người đọc làm sai. Fix comment đúng nghĩa là test xem code và comment nhất quán không.
- **`isinstance` guard tại use site cho dữ liệu qua network/async boundary**: type annotation không runtime-enforced. Dữ liệu từ `asyncio.Task.result()` có thể sai type nếu có bug trong caller. Guard trước khi call `.get()` — fail fast với `None` thay vì `AttributeError`.
- **`_` cho giá trị unpack không dùng**: explicit discard tốt hơn named variable bị bỏ qua. Tránh reader phải scan để xác nhận biến không được dùng.
- **Test coverage là lớp 1, invariant documentation là lớp 2**: hai cơ chế bổ sung không thay thế nhau. Test cover behavior; comment cover constraint không hiển nhiên. Cả hai cần thiết cho codebase bền.

## Glossary mới

- **Regression-resistant code**: code được viết để giảm khả năng bug quay lại sau refactor — qua test coverage (behavior assert), invariant comment (WHY documentation), và defensive guard (validate ở use site).
- **`isinstance` guard at use site**: check `isinstance(x, ExpectedType)` trước khi gọi method của type, ngay tại nơi dùng. Pattern bảo vệ khi type annotation không được enforce hoặc khi dữ liệu đến từ boundary không kiểm soát (network, async, user input).
- **`_` convention**: Python idiom cho "biến được unpack nhưng có chủ ý không dùng". Khác với `__` (name mangling trong class). Dùng trong `for _ in range(n)`, `_, result = some_tuple()`, `done, _ = asyncio.wait(...)`.
