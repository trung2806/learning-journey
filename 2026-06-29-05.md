## Context

Không có horizon item từ D44. User tự đề xuất và viết filter guard mới: khi node trả về error data (unreachable/timeout/bad_data), bỏ qua `tick()` để bảo toàn cooldown state của lần poll thành công trước đó.

## Vấn đề cũ (trước D45)

Khi node bị unreachable, `safe_poll_node` trả về fallback dict với zeros (cpu=0, memory=0, temp=0). `tick()` vẫn được gọi với data zeros này. Hệ quả:
- CPU/memory metrics đều dưới ngưỡng → không có alert → OK
- Nhưng nếu ngưỡng thấp và 0 vẫn trigger alert, daemon sẽ spam false alerts
- Quan trọng hơn: `state_registry[host] = next_state` overwrite state cũ bằng state từ data zeros → cooldown counter bị reset → khi node phục hồi, daemon có thể re-alert ngay thay vì tôn trọng cooldown đang chạy

## Thay đổi kỹ thuật

### 1. `_INFRA_ERROR_STATUSES` constant

```python
_INFRA_ERROR_STATUSES = frozenset({"unreachable", "bad_data", "error", "timeout"})
```

`frozenset` thay vì `set` vì:
- `frozenset` là immutable → hashable → có thể dùng làm dict key hay set element (không cần ở đây nhưng là convention tốt cho module-level constant)
- Tìm kiếm `in frozenset` O(1) — bằng `set`
- Rõ ràng hơn về intent: đây là constant, không phải collection mutable

### 2. Filter guard trong inner loop

```python
# INFRASTRUCTURE FILTER GUARD: bỏ qua node lỗi để bảo toàn cooldown state cũ
healthz = raw_data.get("healthz")
if healthz and healthz.get("status") in _INFRA_ERROR_STATUSES:
    logging.warning(
        f"⚠️ [{host}] Bỏ qua phân tích metrics chu kỳ này do lỗi hạ tầng: "
        f"{healthz['status']} - {healthz.get('reason', 'unknown')}"
    )
    continue
```

**Tại sao `healthz.get("status") in _INFRA_ERROR_STATUSES` thay vì `if healthz:`?**

User viết `if healthz:` — đây là bug do thiếu suy nghĩ về test fixtures. Trace qua 3 cases:
- **Real device data** (không có `healthz` key): `raw_data.get("healthz")` → `None` → `if None and ...` → False → proceed. Đúng.
- **Test fixture** `_HEALTHY_DATA` với `{"healthz": {"status": "healthy"}}`: `if healthz:` → True → `healthz['reason']` → **KeyError** (không có "reason" key) → **crash daemon**!
- **Error fallback** với `{"status": "timeout", "reason": "..."}`: `if healthz:` → True → log → continue. Đúng.

`if healthz:` hoạt động đúng trong production (real device data không có `healthz`) nhưng **crash với test fixtures** vì `KeyError`. 

`healthz.get("status") in _INFRA_ERROR_STATUSES` hoạt động đúng cho cả 3 cases: "healthy" không có trong set → proceed; `None` (no key) không trong set → proceed; "timeout"/"unreachable"/... trong set → filter.

`healthz.get('reason', 'unknown')` trong log message — defensive: dùng `get` với default thay vì `healthz['reason']` để tránh KeyError nếu có data path không có "reason".

## Ruling

- **Filter guard phải check status explicitly, không check `if healthz:`** vì `healthz` key có thể có trong test fixtures với non-error status. Dùng allowlist `_INFRA_ERROR_STATUSES` — chỉ skip khi status rõ ràng là lỗi.
- **`frozenset` cho module-level constant set.** Immutable, hashable, rõ intent về tính không đổi. O(1) lookup như `set`.
- **Bảo toàn cooldown state khi node lỗi.** Không gọi `tick()` với fallback zeros → không update `state_registry[host]` → state từ lần poll cuối thành công được giữ nguyên → cooldown counter không bị reset → khi node phục hồi, behavior đúng.
- **`patch("main.tick", return_value=...)` là pattern spy chuẩn để verify call count.** `wraps=` khi muốn gọi real function nhưng track calls. Không `wraps` khi muốn mock hoàn toàn.
- **`healthz.get('reason', 'unknown')` trong log message.** Defensive: không assume key tồn tại trong dict mình không kiểm soát.

## Glossary mới

- **Infrastructure Filter Guard**: check trước khi gọi `tick()` — nếu `healthz.status` thuộc error set, skip cycle này cho node đó. Preserve state từ lần poll thành công cuối cùng.
- **`frozenset`**: immutable set trong Python. Dùng làm module-level constant khi không cần mutation. `frozenset({"a", "b"})` tạo từ iterable. `in frozenset` O(1) như `set`.
- **Allowlist pattern**: kiểm tra membership trong tập hợp giá trị CỤ THỂ thay vì check boolean. Rõ ràng về ý định và an toàn hơn khi có giá trị ngoài spec (như "healthy" trong test fixture).
