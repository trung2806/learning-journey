## Thay đổi kỹ thuật

### 1. `tasks` từ `list` → `Set[asyncio.Task]`

```python
# TRƯỚC
tasks = [asyncio.create_task(_wrapped_poll(host, poll_timeout)) for host in nodes]
stop_task = asyncio.create_task(_stop_event.wait())
# ... gọi asyncio.wait(tasks + [stop_task], ...) mỗi vòng — list concatenation O(n)

# SAU
tasks: Set[asyncio.Task] = {asyncio.create_task(_wrapped_poll(host, poll_timeout)) for host in nodes}
stop_task = asyncio.create_task(_stop_event.wait())
tasks.add(stop_task)  # stop_task sống trong cùng set
# ... gọi asyncio.wait(tasks, ...) — truyền thẳng set, không copy
```

`asyncio.Task` kế thừa từ `asyncio.Future` vốn implement `__hash__` và `__eq__` theo identity → hashable → có thể lưu trong `set`. Không có gì đặc biệt; đây là behavior chuẩn của Python futures.

### 2. `tasks.discard(fut)` thay `tasks.remove(fut)`

```python
for fut in done:
    if fut is not stop_task:
        tasks.discard(fut)  # O(1), không raise nếu không có
```

`discard` vs `remove`:
- `remove`: O(1) cho set (đã đúng), nhưng raise `KeyError` nếu phần tử không có
- `discard`: O(1), im lặng nếu không có

Vì `asyncio.wait` đảm bảo `done ⊆ tasks` (trả về tập con của tasks đã pass vào), `fut` chắc chắn có trong `tasks` khi vào vòng `for fut in done`. Về mặt lý thuyết `remove` và `discard` tương đương ở đây. `discard` được chọn vì defensive — nếu có bug tương lai khiến `fut` không trong `tasks`, `discard` không làm crash thay vì `remove` raise `KeyError`.

### 3. Loop condition: `while len(tasks) > 1`

```python
# TRƯỚC (list)
while tasks and not _stop_event.is_set():
# ↑ tasks.remove(fut) → tasks rỗng khi hết node → exit

# SAU (set với stop_task embedded)
while len(tasks) > 1 and not _stop_event.is_set():
# ↑ stop_task luôn trong tasks → tasks không bao giờ về 0 cho đến khi discard
# len > 1: còn ít nhất 1 network task ngoài stop_task
```

`len(tasks) > 1` chính xác hơn về ngữ nghĩa: "còn network tasks cần đợi". `tasks and not _stop_event` cũng sẽ work nhưng không capture intent rõ.

**Edge case:** nếu `nodes = []` (không có node nào), `tasks = {stop_task}` → `len = 1` → inner while không chạy. `finally`: `tasks.discard(stop_task)` → `tasks = set()` → `orphaned = []`. Clean. Nhưng `load_nodes` đã raise `ValueError` cho empty list nên case này không xảy ra trong production.

### 4. `finally`: `tasks.discard(stop_task)` trước khi tính orphaned

```python
finally:
    tasks.discard(stop_task)   # tách stop_task khỏi set trước
    if not stop_task.done():
        stop_task.cancel()
        try:
            await stop_task
        except asyncio.CancelledError:
            pass

    orphaned = [t for t in tasks if not t.done()]  # tasks giờ chỉ còn network tasks
    for t in orphaned:
        t.cancel()
    if orphaned:
        await asyncio.gather(*orphaned, return_exceptions=True)
```

Tại sao `discard` trước: nếu không tách, `orphaned` list có thể bao gồm `stop_task` (khi stop_task chưa done). Sau đó `for t in orphaned: t.cancel()` cancel `stop_task` → `await asyncio.gather(*orphaned)` raise `CancelledError` từ stop_task nếu không có `return_exceptions=True`... thực ra với `return_exceptions=True` thì không crash, nhưng ý nghĩa rõ hơn khi tách: stop_task có lifecycle riêng (cancel + try/await), network tasks có lifecycle chung (gather).

### 5. `fut is not stop_task` thay `fut in tasks`

```python
# TRƯỚC
for fut in done:
    if fut in tasks:           # kiểm tra membership → O(1) cho set, nhưng intent mờ
        tasks.remove(fut)

# SAU
for fut in done:
    if fut is not stop_task:   # rõ intent: skip stop_task, xử lý network tasks
        tasks.discard(fut)
```

`fut is not stop_task` dùng identity check (`is`), tương đương với `fut != stop_task` cho objects vì `Task.__eq__` là identity. Intent rõ hơn: "chỉ xử lý các task không phải stop_task".

### 6. Sleep phase: set literal

```python
done_sleep, pending_sleep = await asyncio.wait(
    {sleep_task, stop_task_sleep, reload_task},  # set literal thay list
    return_when=asyncio.FIRST_COMPLETED
)
```

Minor: set literal `{a, b, c}` thay list `[a, b, c]`. `asyncio.wait` nhận bất kỳ iterable nào. Set literal nhất quán với pattern đã dùng cho `tasks` ở trên.

## Ruling

- **`asyncio.Task` là hashable → có thể dùng làm dict key hoặc set element.** Không cần convert sang `id()` hay wrapper. Đây là property của `asyncio.Future` (parent class), documented và stable.
- **Khi `stop_task` nằm trong cùng set với network tasks, loop condition cần đổi.** `while tasks` → `while len(tasks) > 1` vì `tasks` không bao giờ về `set()` cho đến khi `discard(stop_task)` trong `finally`. Mô hình: embedded sentinel = luôn 1 element tồn tại trong set.
- **`discard` phòng thủ hơn `remove` cho set.** Khi `asyncio.wait` đảm bảo membership, cả hai đều đúng. Chọn `discard` cho code mà concurrent modification bug về sau có thể làm `fut` không trong `tasks`.
- **`fut is not stop_task` (identity guard) rõ intent hơn `fut in tasks` (membership check).** Cả hai đúng, nhưng identity guard nói thẳng "đây là sentinel task, không xử lý".

## Glossary cập nhật

- **Set-based Task Registry (D43)**: lưu `asyncio.Task` objects trong `set`, bao gồm cả sentinel task (`stop_task`). Điều kiện thoát: `len(tasks) > 1` (còn ít nhất 1 non-sentinel task). Cleanup: `discard(sentinel)` trước khi tính orphaned. Pattern này tránh `tasks + [stop_task]` concatenation mỗi vòng.
- **`discard` vs `remove` trên set**: `discard` im lặng khi phần tử không có; `remove` raise `KeyError`. Chọn `discard` khi muốn defensive, `remove` khi muốn crash-fast để phát hiện bug.
